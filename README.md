# QA Report Corporate

QA Report Corporate — корпоративная версия редактора QA-чек-листов,
предназначенная для использования в корпоративном контуре с ограниченным и
безопасным набором возможностей.

QA Report — web-инструмент для создания, ведения и публикации QA-чек-листов.
Он помогает быстро собрать проверочный отчёт, импортировать таблицы из Jira,
добавить фактические результаты, скриншоты, файлы и код, а затем отправить
готовый отчёт обратно в Jira или сохранить его в истории.

README описывает текущее состояние проекта. Если меняется сценарий, API,
настройка, ограничение или способ запуска, этот файл нужно обновлять в том же
изменении.

## Назначение версии

- Эта версия отделена от личной версии QA Report и развивается как самостоятельная корпоративная сборка.
- Функциональность может быть урезана в соответствии с требованиями безопасности корпоративного контура.
- Запрещено автоматически возвращать функциональность, предназначенную только для личной версии (`personal-only`).
- Внешние интеграции, хранение токенов, облачные возможности и любые сетевые взаимодействия можно добавлять только после отдельного согласования.

## Корпоративные ограничения

- Не хранить в репозитории секреты, токены, пароли и cookie.
- Не добавлять внешние облачные интеграции без явного запроса и согласования.
- Не восстанавливать функциональность личной версии, если она была удалена из корпоративной.
- Опасный backend-функционал должен быть удалён либо защищён серверными проверками; скрытия только на уровне UI недостаточно.
- Любые изменения, влияющие на безопасность, должны быть явно описаны в MR.

## Разрешённый функционал

Базово в корпоративной версии допускаются:

- редактор чек-листов;
- импорт и экспорт, если они разрешены требованиями корпоративного контура;
- локальное хранение;
- интеграции, отдельно разрешённые для корпоративного использования;
- безопасная история отчётов, если она включена в корпоративной версии.

Описанные ниже возможности не считаются автоматически разрешёнными: их
доступность определяется конфигурацией и требованиями безопасности конкретного

## Пользовательские Возможности

### Редактор чек-листов

- Несколько разделов в одном отчёте.
- Добавление, удаление, дублирование и перемещение строк.
- Создание нового раздела из выбранной строки.
- Пользовательские колонки в каждом разделе.
- Переименование, перемещение и изменение ширины колонок.
- Стабильные `sectionId`, `rowId`, `columnId` для сохранения и сравнения версий.
- Служебные колонки с номером строки и статусом.
- Автоматическая сводка по статусам.

### Статусы проверок

Поддерживаются:

- `OK`
- `НЕ ОК`
- `ПОЧТИ ОК`
- `НЕ ПРОВЕРЕНО`
- `ЧАСТИЧНО ПРОВЕРЕНО`
- `ТРЕБУЕТ УТОЧНЕНИЯ`

Сводка пересчитывается автоматически. Пустые строки не учитываются в итоговой статистике.

### Текст, код, изображения и файлы

- Форматированный вводный текст.
- Жирный, курсив, подчёркивание, зачёркивание, списки, ссылки и цвет текста.
- Кодовые блоки в ячейках и вводном тексте.
- Полноэкранное редактирование кода.
- Копирование кода с Jira-обёрткой `{code}...{code}`.
- Вставка изображений из файлов и буфера обмена.
- Просмотр изображений в увеличенном виде.
- Перетаскивание изображений и кодовых блоков между ячейками.
- Вставка файлов-карточек.

### Импорт и экспорт

- Импорт Jira Wiki Markup вручную.
- Импорт по ссылке на Jira-комментарий.
- Импорт из QA Assistant через API.
- Копирование отчёта как Jira Wiki Markup.
- Копирование визуальной таблицы.
- Предпросмотр Jira-разметки и визуального результата.
- Редактирование Jira-разметки в preview и применение обратно в таблицу.
- Экспорт в XLSX.

### Jira

- Проверка подключения к Jira.
- Публикация отчёта как комментария.
- Поддержка Jira Data Center / Server через Jira Wiki Markup.
- Поддержка Jira Cloud через Atlassian Document Format.
- Загрузка вложений в Jira.
- Диагностика нестандартных ответов Jira, WAF и browser challenge.

### История и версии

- Автосохранение текущего чек-листа.
- Полное локальное хранение в браузере по умолчанию.
- Опциональная серверная история в SQLite, включается во вкладке "Настройки -> История".
- Короткий публичный ID экземпляра, например `851a9bd2`.
- URL текущего чек-листа: `/report/<publicId>`.
- Модалка истории с поиском, источником версии, public ID и комментарием.
- Отдельный пользовательский комментарий для каждого экземпляра отчёта.
- Удаление локальных и серверных записей из истории.
- Очистка истории текущего пространства.

### Конфликты версий

- Обнаружение отличий между локальной и облачной версией.
- Облачные конфликты и синхронизация работают только если включён флаг "Сохранять чек-листы в облачном хранилище".
- Сравнение по содержимому, а не по служебным полям вроде `updatedAt` или `revision`.
- Diff по стабильным ID разделов, строк и колонок.
- Подсветка новых строк, колонок, разделов и изменённых ячеек.
- Защита от перезаписи пользовательских названий колонок дефолтным или пустым значением.
- Выбор облачной версии.
- Явная перезапись облака локальной версией.
- Сохранение обеих версий: облачная остаётся на старом `publicId`, локальная расходящаяся версия создаётся как новая копия с новым `publicId`.

## Быстрый Запуск

Локально:

```bash
npm start
```

Откройте:

```text
http://localhost:4173
```

Эквивалентный запуск без npm:

```bash
node server.js
```

Простое открытие `index.html` подходит только для автономного редактирования.
Jira, загрузка вложений, серверная история, входящий импорт и healthcheck
требуют запущенный `server.js`.

## Основные Сценарии

### Создать чек-лист

1. Откройте приложение.
2. Заполните ссылку на Jira-задачу, окружение и итоговый статус.
3. Добавьте разделы, строки и колонки.
4. Заполните проверки, ожидаемый и фактический результат.
5. При необходимости добавьте изображения, файлы или код.
6. Скопируйте отчёт, экспортируйте XLSX или отправьте результат в Jira.

Новый чек-лист получает случайный короткий `publicId` и URL вида:

```text
/report/851a9bd2
```

### Открыть конкретный чек-лист

Перейдите по URL:

```text
https://qa-report.company.ru/report/851a9bd2
```

Если ID найден в локальном кэше или, при включённом облаке, в серверной истории текущего пространства,
приложение откроет нужный чек-лист. Если ID неизвестен, будет показано понятное
сообщение с возможностью создать новый чек-лист.

Старый формат `/#/checklists/<id>` поддерживается как fallback.

### Использовать историю

В модалке "История отчётов" можно:

- искать отчёты;
- видеть количество всех, облачных и локальных записей;
- видеть номер задачи или пометку «Без задачи»;
- видеть короткий public ID, дату и время сохранения;
- открыть сохранённую версию;
- создать копию;
- удалить запись;
- оставить комментарий к конкретному экземпляру.

Комментарий хранится отдельно от содержимого чек-листа и не влияет на `contentHash`,
поэтому сам по себе не вызывает конфликт версий.

### Разрулить конфликт

Если облачное хранилище включено и облачная версия изменилась, под хедером появляется уведомление
"Есть изменения в облаке" с кнопкой "Сравнить".

В модалке сравнения можно:

- открыть облачную версию;
- оставить локальную и явно перезаписать облако;
- сохранить обе версии.

При сохранении обеих версий исходная облачная запись остаётся канонической и
сохраняет старый `publicId`. Локальная расходящаяся версия сохраняется как новая
копия с новым `publicId`; если пользователь выбирает открыть её, URL сразу
переключается на новый `/report/<id>`.

## Импорт Чек-листов

### Ручной импорт Jira Wiki Markup

В меню Jira есть пункт "Загрузить из Jira".

Поддерживается:

- вставка Jira Wiki Markup напрямую;
- несколько таблиц в одном комментарии;
- разделы `h2.` / `h3.`;
- разные наборы колонок в разных таблицах;
- Jira-цвета статусов;
- кодовые блоки Jira;
- изображения по имени вложения, если они приходят из Jira-комментария вместе со списком вложений.

### Импорт по ссылке на комментарий Jira

Можно вставить прямую ссылку на Jira-комментарий. Приложение:

1. использует текущие настройки Jira;
2. читает комментарий через Jira REST API;
3. подтягивает список вложений задачи, если хватает прав;
4. парсит Wiki Markup для Jira Data Center / Server или ADF для Jira Cloud;
5. создаёт чек-лист в текущем браузере.

### Входящий импорт из QA Assistant

QA Assistant может отправить чек-лист в QA Report через API:

```bash
curl -X POST http://localhost:4173/api/checklists/import \
  -H 'Content-Type: application/json' \
  -d '{
    "source": "qa-assistant",
    "format": "jira",
    "title": "Экспорт транзакций в Excel",
    "issueKey": "https://company.atlassian.net/browse/ADVINTAUT2-117",
    "content": "||Номер||Проверка||Как проверить||Ожидаемый результат||Фактический результат||Статус||Комментарий||\n|1.|Отображение кнопки Export|Авторизоваться как Admin|Кнопка Export видна и доступна для нажатия| | | |"
  }'
```

Поля:

| Поле | Обязательное | Описание |
| --- | --- | --- |
| `format` | да | Пока поддерживается только `jira` |
| `content` | да | Jira Wiki Markup таблицы или чек-листа |
| `source` | нет | Источник, например `qa-assistant` |
| `title` | нет | Название импортируемого чек-листа или первого раздела |
| `issueKey` | нет | Для текущего сценария сюда передаётся полная ссылка на Jira-задачу |

Несмотря на имя `issueKey`, для входящего импорта поле используется как ссылка на
задачу. Передавайте полный URL:

```text
https://company.atlassian.net/browse/ADVINTAUT2-117
```

Успешный ответ:

```json
{
  "ok": true,
  "checklistId": "ec8d6464-70e0-4c84-ba6e-31fbd7fd1c9f",
  "publicId": "851a9bd2",
  "url": "https://qa-report.company.ru/report/851a9bd2?importToken=ec8d6464-70e0-4c84-ba6e-31fbd7fd1c9f",
  "expiresAt": 1783285561851,
  "parsed": {
    "sections": 1,
    "rows": 15
  }
}
```

Дальше нужно открыть `url` в браузере. По этой ссылке QA Report заберёт payload
с сервера, создаст новый чек-лист и очистит `importToken` из адресной строки.

Публичный origin для `url` берётся из `QA_REPORT_PUBLIC_URL`, `APP_PUBLIC_URL`
или `PUBLIC_URL`, если переменная задана. Если переменной нет, сервер использует
`Forwarded` или `X-Forwarded-Proto` + `X-Forwarded-Host`; без них будет использован
обычный `Host` запроса, например `localhost:4173` при локальном вызове.

Ошибки:

| Код | Когда возвращается |
| --- | --- |
| `400` | `format` не равен `jira` |
| `400` | `content` пустой или не строка |
| `422` | Jira-разметку не удалось распарсить в пункты чек-листа |
| `404` | import token не найден или истёк |

Payload входящего импорта хранится в SQLite 15 минут и удаляется после истечения TTL.
Jira-разметка не передаётся через query string; в URL находится только короткий
import token.

## Интеграция С Jira

В интерфейсе есть настройки Jira и действие "Отправить в Jira".

### Поддерживаемые варианты

| Jira | Формат комментария | Авторизация |
| --- | --- | --- |
| Jira Data Center / Server | Jira Wiki Markup | Personal Access Token, Basic Auth или Cookie |
| Jira Cloud | Atlassian Document Format | Email Atlassian + API token |

Адрес Jira, тип Jira и пользователь сохраняются локально в браузере. Токен или
пароль не пишутся в `localStorage` и живут только в памяти открытой вкладки.

### Публикация комментария

При публикации приложение:

1. берёт полную ссылку из поля "Ссылка на задачу";
2. извлекает ключ задачи из URL;
3. готовит комментарий в нужном формате;
4. загружает локальные вложения в Jira;
5. создаёт комментарий;
6. проверяет, что комментарий создан и читается обратно.

Если Jira или промежуточный proxy возвращает успешный HTTP-статус без объекта
комментария, приложение пытается найти созданный комментарий среди последних
комментариев задачи. При ошибке показываются диагностические детали: upstream
HTTP-статус, `Location`, краткое тело ответа, количество комментариев до и после
публикации и ошибки контрольного чтения.

### Вложения

Для Jira Data Center / Server:

- приложение читает существующие вложения задачи;
- выбирает свободное имя `screenshot-N.png`;
- вставляет изображения в комментарий как `!screenshot-N.png|thumbnail!`;
- если список вложений недоступен, к имени добавляется временная метка.

Для Jira Cloud:

- вложения публикуются как кликабельные attachment-card;
- код публикуется как ADF `codeBlock`;
- встроенное thumbnail-превью через Media Services UUID не используется.

Перед отправкой приложение проверяет сигнатуру изображения, MIME-тип и расширение.
Файлы загружаются по одному, чтобы проще диагностировать ошибку конкретного вложения.

### WAF / SSO / Browser Challenge

Если корпоративный WAF или SSO возвращает HTML browser challenge вместо JSON Jira
REST API, приложение останавливает публикацию до POST и возвращает
`JIRA_SECURITY_CHALLENGE`.

Для серверной интеграции администратору нужно разрешить исходящий IP приложения
либо исключить `/rest/api/*` из интерактивной browser-проверки при наличии
корректного `Authorization` header.

Cookie браузерной Jira-сессии приложение не копирует и не сохраняет.

## Обратная Связь

Кнопка с иконкой жука сохранена в интерфейсе, но отправка обратной связи временно
недоступна. При нажатии открывается информационное окно. Серверное сохранение,
email-отправка и интеграция с Resend в корпоративной версии удалены.

## Удалённые Внешние Интеграции

Интеграции с Яндекс.Диском и Google Drive полностью удалены из корпоративной
версии: в приложении нет их настроек, OAuth-токенов, клиентских действий и
серверных маршрутов загрузки. Возвращать эти интеграции без отдельного
согласования запрещено.

## Корпоративное Файловое Хранилище

Корпоративная версия поддерживает приватное S3-совместимое хранилище MinIO.
Пользователь не управляет подключением и не видит ключи. Для изображений и
файлов доступно действие «Загрузить в хранилище»; после загрузки Base64 заменяется
ссылкой на защищённый маршрут QA Report.

При включённой серверной истории вложения переносятся в MinIO автоматически до
сохранения отчёта в SQLite. Слева сверху на изображении или файловой карточке
показывается состояние: индикатор загрузки для локального объекта и облако для
объекта, уже сохранённого в MinIO. Превью изображения при автоматической
миграции сохраняется и загружается по удалённой ссылке.

Ручной сценарий отделён от автоматического. До загрузки в контекстном меню есть
пункт «Загрузить в хранилище». После успешной загрузки он заменяется пунктом
«Заменить на ссылку», который по явному выбору пользователя заменяет превью или
файловую карточку обычной кликабельной ссылкой.

При экспорте Jira Wiki Markup объекты MinIO преобразуются в ссылки вида
`[название|URL]`, а не в синтаксис Jira-вложения.

Объекты размещаются по ID отчёта:

```text
reports/<ownerHash>/<reportId>/report.json
reports/<ownerHash>/<reportId>/images/<uuid>-<filename>
reports/<ownerHash>/<reportId>/files/<uuid>-<filename>
```

Настройка выполняется только на сервере:

| Переменная | Назначение |
| --- | --- |
| `QA_STORAGE_ENDPOINT` | HTTPS endpoint MinIO |
| `QA_STORAGE_BUCKET` | Имя корпоративного бакета |
| `QA_STORAGE_REGION` | S3-регион, для MinIO обычно `us-east-1` |
| `QA_STORAGE_ACCESS_KEY_FILE` | Путь к secret-файлу с access key |
| `QA_STORAGE_SECRET_KEY_FILE` | Путь к secret-файлу с secret key |
| `QA_STORAGE_ACCESS_KEY` | Access key для локальной разработки через игнорируемый `.env` |
| `QA_STORAGE_SECRET_KEY` | Secret key для локальной разработки через игнорируемый `.env` |
| `QA_JIRA_ALLOWED_ORIGINS` | Разрешённые Jira origin через запятую; произвольные адреса запрещены |
| `QA_REPORT_TRUST_PROXY` | Разрешает учитывать `Forwarded`/`X-Forwarded-*` только за доверенным proxy |
| `QA_REPORT_TRUST_PROXY_IDENTITY` | Разрешает доверять SSO identity-заголовкам только за доверенным proxy |

Значения ключей нельзя хранить в репозитории или передавать в браузер.
Объекты разделяются хешем владельца или корпоративного пространства, поэтому
одинаковые ID отчётов разных пользователей не дают доступ к чужим файлам.

Для локальной настройки используйте файл `.env` в корне проекта. Он уже создан,
исключён через `.gitignore` и содержит пустые поля для ключей. После изменения
`.env` необходимо перезапустить `npm start`. `.env.example` служит безопасным
шаблоном без секретов.

## Защита Web-интерфейса

- Импортируемая Jira-разметка проверяет URL и разрешает в ссылках только `http` и `https`.
- Форматированный HTML отчётов очищается по списку разрешённых элементов и атрибутов до отображения и сохранения.
- Сервер удаляет исполняемые HTML-элементы и обработчики событий перед записью отчёта в SQLite.
- Сервер устанавливает CSP и защитные HTTP-заголовки, запрещает cross-site изменения и ограничивает частоту API-запросов.

## Хранение Данных

| Данные | Где хранятся |
| --- | --- |
| Текущий черновик | `localStorage` браузера |
| Локальный кэш истории | IndexedDB браузера |
| Серверная история отчётов | SQLite-файл `reports-data/qa-report.sqlite`, используется только если пользователь включил облачное хранилище |
| Public ID экземпляра | SQLite и локальный кэш |
| Комментарий истории | SQLite для облачных записей, IndexedDB для локальных |
| Настройки Jira без секрета | `localStorage` браузера |
| Jira token / пароль | Только память открытой вкладки |
| Изображения и файлы облачных отчётов | MinIO, в SQLite сохраняются ссылки и метаданные |
| Входящий импорт QA Assistant | SQLite, автоматически удаляется через 15 минут |

По умолчанию чек-листы сохраняются только локально в браузере. Если пользователь
включил "Сохранять чек-листы в облачном хранилище", локальные Base64-вложения
сначала загружаются в MinIO. После этого серверная история хранит в
`document_json` документ со ссылками и метаданными, а `report.json` сохраняется в
директории отчёта в MinIO. Бинарные данные в SQLite не записываются.

Метаданные вынесены в отдельные колонки: название, ссылка на Jira-задачу, ключ
задачи, окружение, общий статус, даты, владелец, `public_id`, `content_hash`,
`history_comment` и `deleted_at`.

Старые записи без `public_id` получают его при миграции или первом чтении.

## Защита От Конфликтов

В интерфейсе используются два понятия:

- локальная версия - копия чек-листа в текущем браузере;
- облачная версия - копия чек-листа в серверной истории SQLite.

При сохранении фронтенд отправляет последний известный `contentHash`. Если
облачная версия уже изменилась и hash не совпадает с базовым hash клиента,
backend возвращает `409 Conflict` и не перезаписывает документ.

`contentHash` считается по нормализованному содержимому чек-листа. В него не входят:

- `updatedAt`;
- `revision`;
- `lastSavedBy`;
- `lastSavedClientId`;
- UI-состояние фокуса, скролла и выбранной ячейки;
- комментарий записи в истории.

Такой механизм защищает от сценария, когда один и тот же отчёт открыт на двух
компьютерах: старая вкладка не сможет молча затереть более новую облачную версию.

## Как Определяется Владелец Истории

QA Report не реализует собственные логины и пароли. Backend выбирает владельца
по доступному контексту:

| Приоритет | Источник | Как работает |
| --- | --- | --- |
| 1 | SSO headers | Если reverse proxy передал `X-Forwarded-Email`, `X-Auth-Request-Email`, `X-Forwarded-User`, `X-Auth-Request-User` или `Remote-User`, отчёты привязываются к корпоративной учётке |
| 2 | Ключ пространства | Если пользователь указал ключ в настройках истории, отчёты привязываются к этому ключу |
| 3 | `clientId` браузера | Если SSO и ключа нет, отчёты привязываются к автоматически созданному ID текущего браузера |
| 4 | Anonymous fallback | Если браузер не передал `clientId`, используется технический fallback по адресу клиента |

Ключ пространства нужен для личного сервера и простого переноса между компьютерами:
укажите одинаковый ключ в настройках истории на разных устройствах, и они увидят
одну серверную историю. Это не полноценная авторизация: кто знает ключ, тот
получает доступ к этому пространству.

В корпоративном контуре рекомендуемый вариант - ставить QA Report за SSO/reverse
proxy и передавать identity через заголовки. Приложение при этом не хранит пароли
пользователей.

## Конфигурация

| Переменная | Назначение |
| --- | --- |
| `PORT` | Порт Node-сервера, по умолчанию `4173` |
| `HOST` | Host для `server.listen`, по умолчанию `127.0.0.1`; в Docker используется `0.0.0.0` |
| `QA_REPORT_PUBLIC_URL` | Публичный origin для ссылок импорта, например `https://qa-report.company.ru` |
| `APP_PUBLIC_URL` / `PUBLIC_URL` | Альтернативные имена переменной публичного origin |
| `REPORTS_DB_PATH` | Путь к SQLite-файлу истории |
| `QA_REPORT_STORE_ATTACHMENTS` | Если `true`, сервер снова будет хранить embedded изображения и file-card в SQLite; по умолчанию выключено |
| `QA_STORAGE_*` | Настройки корпоративного MinIO, описанные выше |
| `QA_JIRA_ALLOWED_ORIGINS` | Разрешённые Jira origin через запятую |
| `QA_REPORT_TRUST_PROXY` | Доверие transport-заголовкам proxy |
| `QA_REPORT_TRUST_PROXY_IDENTITY` | Доверие SSO identity-заголовкам proxy |
| `QA_REPORT_MAX_BODY_MB` | Максимальный размер JSON-запроса к backend, по умолчанию `150` |
| `QA_REPORT_MAX_ATTACHMENT_MB` | Максимальный размер одного вложения Jira, по умолчанию `50` |

Для публичного сервера рекомендуется задать `QA_REPORT_PUBLIC_URL`, даже если
reverse proxy передаёт `Forwarded` или `X-Forwarded-*`. Это убирает риск, что API
импорта вернёт внутренний `localhost` или адрес контейнера.

Если приложение стоит за Nginx/Caddy/Traefik, лимит тела запроса на reverse proxy
должен быть не ниже `QA_REPORT_MAX_BODY_MB`. Для Nginx обычно нужен
`client_max_body_size 150m;` или больше для endpoint `/api/jira/attachments`.

## Docker

Запуск:

```bash
docker compose up -d --build
```

Приложение будет доступно:

```text
http://127.0.0.1:4173
```

Изменить внешний порт можно через `.env`:

```env
APP_PORT=8080
```

Для публичного URL импорта:

```env
QA_REPORT_PUBLIC_URL=https://qa-report.company.ru
```

SQLite-файл серверной истории хранится в volume `reports_data`.

Путь к SQLite можно переопределить:

```env
REPORTS_DB_PATH=/app/reports-data/qa-report.sqlite
```

Для бэкапа достаточно сохранять volume `reports_data` или сам файл
`qa-report.sqlite` вместе с `-wal`/`-shm` файлами, если они есть рядом.

Проверка:

```bash
docker compose ps
curl http://localhost:4173/api/health
```

Логи:

```bash
docker compose logs -f qa-report
```

Остановка:

```bash
docker compose down
```

Публиковать Node.js-порт напрямую запрещено. По умолчанию Compose привязывает его
только к `127.0.0.1`. Для корпоративного развертывания обязателен HTTPS reverse
proxy или ingress (Nginx, Caddy, Traefik) с валидным сертификатом. Флаги доверия
к forwarded- и SSO-заголовкам можно включать только когда прямой доступ к Node.js
закрыт, а proxy перезаписывает эти заголовки.

## Известные Ограничения

- Входящий импорт из QA Assistant хранится временно в SQLite и удаляется после TTL.
- Вложения облачной истории требуют доступного и настроенного MinIO. В SQLite
  сохраняются только ссылки и метаданные.
- Ключ пространства не является строгой авторизацией. Для корпоративной приватности
  нужен SSO/reverse proxy, который передаёт user identity в заголовках.
- SQLite рассчитан на один сервер/контейнер. Для горизонтального масштабирования
  нужен переход на PostgreSQL или другое внешнее хранилище.
- Jira Cloud не получает встроенное thumbnail-превью через Media Services, только
  кликабельные attachment-card.
- Итоговое отображение Jira Wiki Markup зависит от конкретной версии и настроек
  Jira Data Center / Server.
- Корпоративные WAF/SSO могут блокировать серверные Jira REST-запросы.
- Токены Jira не сохраняются между перезагрузками вкладки.
